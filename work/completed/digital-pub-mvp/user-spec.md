# User Spec: Диджитал Паб — MVP

## Что делаем

Курированный job board для диджитал-фрилансеров — d-pub.ru. Агрегирует вакансии и резюме из 10 отобранных Telegram-каналов через бота @vacancy_bot_ren (бот-парсер, собирает вакансии по ключевым словам и кросспостит в канал @web_vacancy). Для подачи вакансий и резюме используется отдельный бот @resume_vac_bot (бот-форма, принимает заявки от пользователей). MVP: полноценный многостраничный сайт с responsive-дизайном, работающими фильтрами, поиском, разделами по тегам, статьями и откликами через Telegram.

**Агенты** — это AI-агенты Claude Code (Marketer, SEO, Writer, Designer, Dev), автоматизирующие создание контента и разработку по заданию владельца проекта.

## Зачем

Вакансии в digital-фрилансе разбросаны по десяткам Telegram-каналов. Фрилансерам сложно мониторить все. Работодателям сложно найти надёжного исполнителя. Диджитал Паб собирает всё в одном месте, фильтрует и организует — с фокусом на качество, а не количество. Человечный подход вместо бездушных hh.ru и Habr Career.

## Как должно работать

### Фрилансер ищет вакансию

1. Заходит на главную — видит ленту вакансий и резюме
2. Использует фильтр-чипы (SMM, SEO, Дизайн, Таргет...) для фильтрации по тегам (Tag/PostTag)
3. Или нажимает тег в правой панели — попадает на SEO-страницу раздела `/vacancies/{tag}` с привязанными постами через Tag/PostTag и SEO-текстом внизу
4. Открывает карточку вакансии — видит полное описание, зарплату, контакт
5. Нажимает «Откликнуться» — переходит в Telegram к работодателю

### Фрилансер размещает резюме

1. Нажимает «+ Разместить» в хэдере или футере
2. Переходит в Telegram-бота @resume_vac_bot
3. Заполняет резюме через бота
4. После модерации резюме публикуется на сайте

### Работодатель размещает вакансию

1. Нажимает «+ Разместить» — переходит в бота @resume_vac_bot
2. Заполняет вакансию через бота
3. После модерации вакансия появляется на сайте

### Работодатель ищет фрилансера

1. Заходит в раздел «Резюме»
2. Фильтрует по специальности
3. Находит подходящего фрилансера
4. Связывается через контакт в резюме

### SEO-статьи

1. Тони даёт ТЗ на статью
2. Агенты (SEO → Marketer → Writer) подбирают семантику и пишут статью
3. Статья сохраняется как draft в markdown
4. Тони читает и утверждает
5. Статья публикуется через API → появляется на `/articles/{slug}`

### Читатель приходит из поиска

1. Пользователь ищет в Google «работа SMM удалённо»
2. Попадает на SEO-статью или SEO-страницу тега `/vacancies/smm`
3. Читает статью / просматривает вакансии
4. Видит релевантные вакансии в сайдбаре или внизу статьи
5. Переходит к конкретной вакансии → откликается через Telegram

## Критерии приёмки

### Основной функционал
- [ ] Главная страница: лента вакансий + резюме из PostgreSQL, пагинация «Показать ещё»
- [ ] Фильтр-чипы на главной фильтруют ленту по тегам (Tag/PostTag связи в БД)
- [ ] Поиск в навбаре ищет по title, description, company
- [ ] Счётчики вакансий/резюме в сайдбаре — реальные из БД (не захардкожены)
- [ ] Посты без описания не показываются в ленте

### Разделы и навигация
- [ ] Навбар: Главная, Вакансии, Резюме, Статьи (убраны Отзывы, Курсы, Полезное)
- [ ] Раздел `/vacancies` — листинг всех вакансий с фильтрацией по категориям
- [ ] Раздел `/resumes` — листинг всех резюме с фильтрацией
- [ ] SEO-страницы по тегам: `/vacancies/{tag-slug}` — отдельные страницы с привязкой постов через Tag/PostTag (M:N связи в БД), уникальный title, meta description, h1, SEO-текст внизу
- [ ] Теги в правой панели ведут на SEO-страницы (не на фильтры главной)
- [ ] Теги динамические — не показываются если 0 привязанных публикаций
- [ ] Синхронизация из Telegram автоматически назначает теги постам при импорте (по ключевым словам → Tag/PostTag)

### Статьи
- [ ] Раздел `/articles` — листинг статей из БД
- [ ] Страница статьи `/articles/{slug}` с полным текстом
- [ ] Статьи хранятся как MDX-файлы в репо с поддержкой батчевого деплоя (3-5 статей за раз)
- [ ] ≥10 статей опубликовано к моменту запуска
- [ ] Страницы /privacy и /terms с базовым текстом (152-ФЗ)
- [ ] sitemap.xml генерируется автоматически (app/sitemap.ts)

### Отклик и подача
- [ ] Кнопка «Откликнуться» на странице вакансии — ведёт в Telegram работодателя
- [ ] Кнопка «+ Разместить» в хэдере — ведёт на бота @resume_vac_bot
- [ ] Ссылки «Разместить вакансию» и «Реклама» в футере — ведут на бота

### Дизайн и адаптивность
- [ ] Полноценный responsive на Tailwind CSS (mobile-first): бургер-меню на экранах <768px, сайдбары скрыты, карточки в 1 колонку
- [ ] Миграция стилей с кастомного CSS на Tailwind CSS — все компоненты переведены на Tailwind-классы
- [ ] Dark/Light тема с анимированным переключением (transition ≥200ms) через Tailwind dark: модификатор
- [ ] На мобильных (320px+): нет горизонтального скролла, текст читаем без зума, кнопки ≥44px touch target

### Инфраструктура
- [ ] Домен d-pub.ru привязан, SSL настроен
- [ ] Автосинхронизация из Telegram по cron
- [ ] E2E тест критического пути проходит
- [ ] Интеграционный тест парсинга Telegram проходит

## Ограничения

- **Команда:** 1 человек + AI (Claude Code)
- **Срок:** 1 месяц на MVP
- **Стек:** Next.js 14, PostgreSQL, Prisma — менять нельзя
- **Хостинг:** VPS на NetAngels, PM2 + Nginx
- **Подача вакансий/резюме:** в MVP только через Telegram-бота (не на сайте)
- **Модерация:** в MVP через Telegram-бота, полноценная админка — в итерации 2
- **Авторизация пользователей:** не в MVP

## Риски

- **Миграция на Tailwind CSS:** 1360 строк кастомного CSS нужно перевести на Tailwind-классы — значительный объём работы. **Митигация:** мигрировать компонент за компонентом, начиная с layout → navbar → feed → sidebars → footer. Mobile-first подход.
- **10 статей за месяц:** плотный график параллельно с разработкой. **Митигация:** агенты (SEO/Writer) автоматизируют создание, от Тони только ТЗ и апрув.
- **Telegram парсинг:** t.me/s/ может заблокировать частые запросы. **Митигация:** мониторинг ошибок через бота @web_vacancy_bot, fallback на mock-данные.
- **SEO-страницы тегов:** без качественных meta и текстов нет SEO-эффекта. **Митигация:** агент SEO генерирует тексты для каждого тега.
- **Права на .next/:** продакшн-процесс (root) и dev-процесс (claude) конфликтуют. **Митигация:** настроить раздельные distDir или единого пользователя.
- **Отсутствие трафика на старте:** новый домен без SEO-истории, органический трафик появится через 3-6 месяцев. **Митигация:** анонс в Telegram-каналах (@web_vacancy, @pub_resume), прямые ссылки из каналов-источников, кросспост в Макс и ВК.

## Технические решения

- Отклик на вакансию — через прямую ссылку в Telegram работодателя, без формы на сайте в MVP.
- Подача вакансий/резюме — через Telegram-бота @resume_vac_bot, кнопка «Разместить» ведёт туда.
- Статьи хранятся как MDX-файлы в репо (`app/articles/`). Агенты создают файлы пачками (3-5 штук), коммитят — деплой подхватывает. Поддержка батчевого деплоя: скрипт обрабатывает все новые MDX-файлы за раз. API для статей — в итерации 2 вместе с админкой.
- Теги и фильтрация — через реальные Tag/PostTag связи в БД (M:N). Синхронизация из Telegram автоматически назначает теги постам по ключевым словам при импорте.
- SEO-страницы тегов (`/vacancies/{tag}`) — отдельные страницы с фильтрацией через Tag/PostTag связи в БД, не текстовый поиск. Каждая страница имеет уникальный title, meta, h1, SEO-текст.
- Навбар сокращён до 4 пунктов (Главная, Вакансии, Резюме, Статьи) — остальные разделы не в MVP.
- Статистика платформы (счётчики) — реальные запросы к БД, не захардкоженные числа.
- Responsive — полная миграция на Tailwind CSS (mobile-first). Кастомный CSS в globals.css заменяется Tailwind-классами. CSS-переменные для темы сохраняются, но стилизация компонентов — через Tailwind.
- Синхронизация из Telegram пишет посты со статусом `published` напрямую, без модерации. Модерация — в итерации 2 с админкой.
- **Дизайн:** перед вёрсткой агент Designer создаёт HTML-макеты в песочнице (статический сервер на отдельном порту). Макеты утверждаются Тони, затем Dev верстает по ним. Workflow: user-spec → Designer → макеты → апрув → tech-spec → Dev.

## Тестирование

**Unit-тесты:** делаются всегда, не обсуждаются.

**Интеграционные тесты:** делаем — тест парсинга Telegram (скрипт sync-telegram.ts корректно извлекает посты и сохраняет в БД).

**E2E тесты:** делаем — критический путь: главная → фильтр → вакансия → кнопка «Откликнуться» ведёт в TG. Плюс: проверка responsive на мобильном viewport.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Открыть главную | curl http://localhost:3100 | HTTP 200, HTML содержит карточки вакансий |
| 2. Проверить фильтрацию | playwright | Клик на чип «SMM» → в ленте только посты с «SMM» |
| 3. Открыть SEO-страницу тега | curl /vacancies/smm | HTTP 200, уникальный title и h1, SEO-текст внизу |
| 4. Проверить отклик | playwright | Кнопка «Откликнуться» содержит href на t.me/ |
| 5. Проверить статьи | curl /articles | HTTP 200, список статей из БД |
| 6. Проверить статью | curl /articles/{slug} | HTTP 200, MDX-статья рендерится |
| 7. Проверить responsive | playwright (viewport 375px) | Бургер-меню, 1 колонка, нет горизонтального скролла |
| 8. Проверить sync | bash: npm run sync | Новые посты появляются в БД |
| 9. Проверить счётчики | curl / + grep | Числа в сайдбаре соответствуют COUNT(*) из БД |

### Пользователь проверяет

- Открыть сайт на телефоне — убедиться что навигация, карточки, фильтры удобны
- Нажать «Откликнуться» на вакансии — убедиться что Telegram открывается с правильным контактом
- Нажать «+ Разместить» — убедиться что переход на бота @resume_vac_bot
- Проверить dark/light тему — плавное переключение, все элементы читаемы
- Прочитать SEO-статью — убедиться что текст качественный и форматирование корректно
