## 3. Модель данных (Prisma)

**Файл**: `prisma/schema.prisma`

### Таблицы:

#### Post (основная)
| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, autoincrement, PK | - |
| type | PostType (vacancy / resume) | Тип поста |
| title | String | Заголовок |
| slug | String?, unique | ЧПУ-адрес |
| description | String? | Полный текст |
| company | String? | Компания |
| salary | String? | Зарплата (текст) |
| status | PostStatus (pending / published / rejected) | Статус модерации, default: pending |
| source | PostSource (telegram / user) | Источник, default: telegram |
| imageUrl | String? | Путь к изображению |
| telegramMessageId | String? | ID сообщения в TG |
| channelUsername | String? | Username канала |
| categoryId | Int? → Category | FK на категорию |
| createdAt, updatedAt | DateTime | Временные метки |

Уникальный композитный индекс: `[telegramMessageId, channelUsername]`

#### Tag
| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, PK | - |
| name | String, unique | Название тега |
| tagType | TagType (format / level / specialization) | Тип тега |

#### PostTag (связка M:N)
| Поле | Тип | Описание |
|------|-----|----------|
| postId | Int → Post | FK |
| tagId | Int → Tag | FK |

Композитный PK: `[postId, tagId]`

#### Category
| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, PK | - |
| name | String, unique | Название |
| slug | String, unique | ЧПУ |

#### Article
| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, PK | - |
| title | String | Заголовок |
| slug | String, unique | ЧПУ |
| content | String | Содержимое |
| publishedAt | DateTime? | Дата публикации |
| createdAt, updatedAt | DateTime | Временные метки |

### Связи:
- Post → Category (M:1, через categoryId)
- Post ↔ Tag (M:N, через PostTag)

### Статус использования моделей:
- **Post** — активно используется (sync, feed, detail page)
- **Tag, PostTag** — схема создана, но теги НЕ назначаются при синхронизации и НЕ отображаются в UI
- **Category** — схема создана, но categoryId НЕ проставляется при синхронизации; фильтрация по категориям реализована через текстовый поиск в title/description
- **Article** — схема создана, но НЕТ ни одного API/страницы для статей; данные не создаются

### Миграции:
- `20260407110907_init` — начальная
- `20260501084837_add_image_url` — добавление imageUrl
- `20260501090441_add_slug` — добавление slug

## 4. Синхронизация с Telegram

### Основной скрипт: `scripts/sync-telegram.ts`
- Запуск: `npm run sync` (через tsx)
- Парсит HTML страницы `t.me/s/{channel}` (публичный preview)
- Каналы: `web_vacancy` (вакансии), `pub_resume` (резюме)
- Извлекает: текст, изображения (background-image из CSS), messageId
- Генерирует slug через транслитерацию кириллицы
- Парсит зарплату регулярным выражением
- Скачивает изображения в `public/images/posts/`
- Сохраняет в БД со статусом `published` (автопубликация, без модерации)
- Дедупликация по `[telegramMessageId, channelUsername]`

**Статус**: Реализовано

### Скрипт скачивания фото: `scripts/download-photos.py`
- Использует Telethon (Python) для получения фото в полном качестве
- Запускается в Docker-контейнере бота (`ghcr.io/antonrendel-dev/telegram-bot-vac`)
- Перезаписывает превью-изображения полноразмерными

**Статус**: Реализовано

### Автосинхронизация: `scripts/auto-sync.sh`
- Предназначен для cron: `*/30 * * * *`
- Шаг 1: sync-telegram.ts
- Шаг 2: download-photos.py через Docker
- Шаг 3: `npm run build` + `pm2 restart digital-pub`
- Логирование в `logs/sync.log`

**Статус**: Реализовано

## 10. Реализовано vs Задекларировано

### Реально работает:
- Главная лента с вакансиями и резюме из PostgreSQL
- Страницы /vacancies и /resumes с фильтрацией по категориям (текстовый поиск)
- Детальная страница поста (/post/[id] и /vacancies/[slug])
- Фильтр-чипы в ленте (клиентская фильтрация по title/description)
- Поиск в навбаре (клиентская фильтрация)
- Синхронизация постов из Telegram-каналов
- Dark/Light тема с плавным переключением
- Пагинация "Показать ещё" (по 10 штук)
