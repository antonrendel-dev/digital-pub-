# Tech Spec: Payload CMS Integration

## Solution

Two-phase implementation. **Phase 1:** upgrade Next.js 14.2.35 → 15 and React 18 → 19 as an isolated, independently-deployable change — all dynamic route params (`params`, `searchParams`) become `Promise`-typed and must be awaited. **Phase 2:** install Payload CMS v3, define Collections (Tags, Posts, Articles, Media) and Globals (Navbar, Footer, SocialChannels, SiteSettings), run one-time data migration scripts moving existing Prisma tables to Payload-managed tables, replace the Prisma data-access layer (`lib/tags.ts`, `lib/posts.ts`, `app/sitemap.ts`) with Payload Local API, wire `afterChange` hooks to call `revalidatePath()` directly (same process — no HTTP), and update `scripts/sync-telegram.ts` on the red server to write to Payload via its REST API.

MDX articles (`content/articles/`) stay in v1. The Payload `articles` collection is created for new agent-published content; the articles listing merges both sources sorted by date.

Prisma is fully removed after data migration is confirmed on staging.

## Architecture

### What we're building/modifying

- **`payload.config.ts`** — Payload configuration root: imports from `payload/collections/*` and `payload/globals/*`; `db-postgres` adapter with explicit `url: process.env.DB_CONNECTION_STRING`; `richtext-lexical` for rich text fields; `onInit` hook for admin bootstrap

- **`app/(payload)/admin/[[...segments]]/page.tsx`** — Payload admin UI route (auto-generated by Payload's Next.js adapter)
- **`middleware.ts`** — protects `/admin` routes; redirects unauthenticated requests to `/admin/login`
- **`lib/tags.ts`** — replace Prisma queries with Payload Local API calls
- **`lib/posts.ts`** — replace Prisma queries with Payload Local API calls
- **`lib/postUtils.ts`** — types-only, no ORM dependency; update `FeedPost` to match Payload field names
- **`app/sitemap.ts`** — replace Prisma queries with Payload Local API
- **`next.config.mjs`** — wrap with `withPayload()`, add `worker-src blob:` to CSP for `/admin/*`

- **`.github/workflows/deploy.yml`** — remove Prisma generate step, add `payload migrate` SSH step, add `--exclude=public/uploads/`
- **`.github/workflows/deploy-staging.yml`** — same Payload migrate and rsync changes for staging

### How it works

**Admin save → page revalidation:**

```
Admin saves Tag/Post/Article/Global in Payload UI
  → Payload afterChange hook fires (same Node.js process as Next.js)
  → revalidatePath() called with specific paths
  → Next.js cache invalidated
  → Next request serves fresh HTML (≤3 sec for content, ≤1 sec for Globals)
```

**Agent publishes article:**

```
Agent sends POST /api/articles (custom Next.js API route)
  with Authorization: users API-Key <token>
  → Route extracts token, calls payload.auth({ headers })
  → Checks req.user.role === 'agent', then payload.create({ collection: 'articles', ... })
  → afterChange hook fires → revalidatePath('/articles', 'layout')
  → Article appears on site within ≤3 sec
```

**Telegram sync (red server → Payload REST API):**

```
Cron on red server runs scripts/sync-telegram.ts
  → Fetch /api/tags?limit=200 → build slug→id map
  → Parse Telegram channels
  → For each new post: POST /api/posts with tag IDs
    → On 409 conflict (unique constraint): skip (dedup)
  → Posts appear on d-pub.ru/vacancies after next ISR revalidation
```

**Next.js 15 async params:**

```
Old (Next.js 14):  const { category } = params
New (Next.js 15):  const { category } = await params
Affects: 5 dynamic route files + generateMetadata in each
```

## Decisions

### Decision 1: Payload Local API for in-process data access

**Decision:** `lib/tags.ts`, `lib/posts.ts`, and `app/sitemap.ts` use `getPayload()` + Local API (`payload.find()`, `payload.findByID()`) — not Payload's HTTP REST API.  
**Rationale:** Payload runs in the same Next.js process. Local API = direct function calls, no HTTP overhead, no auth token needed, typed return values. Supports US-1 (instant tag SEO edits), US-2 (articles listing).  
**Alternatives considered:** Fetch Payload's auto-generated REST `/api/*` — adds unnecessary HTTP round-trip within same process.

### Decision 2: afterChange hooks call revalidatePath() directly

**Decision:** Payload collection/global hooks import `revalidatePath` from `'next/cache'` and call it synchronously in the hook body. No separate `/api/revalidate` HTTP endpoint.  
**Rationale:** Payload and Next.js share the same Node.js process (via `withPayload()`). `revalidatePath()` works in server context — hooks run in that context. Supports US-1, US-3 revalidation SLA requirements (≤3 sec content, ≤1 sec Globals).  
**Alternatives considered:** Separate `/api/revalidate` endpoint with secret header — adds a network hop and extra authentication layer with no benefit in same-process architecture.

### Decision 3: revalidatePath strategy per collection

**Decision:**

- Tag saved → `revalidatePath('/vacancies/[category]', 'page')` with tag's slug + `revalidatePath('/', 'layout')`
- Post saved → `revalidatePath('/vacancies/[category]', 'page')` for its primary category + `revalidatePath('/vacancies', 'layout')`
- Article saved → `revalidatePath('/articles', 'layout')` + `revalidatePath('/articles/[slug]', 'page')` with article slug
- Any Global saved → `revalidatePath('/', 'layout')` (invalidates all pages sharing root layout)

**Rationale:** Targeted invalidation avoids full-site cache bust on every post save. Global change invalidates all pages since Navbar/Footer are in root layout. Supports user-spec revalidation SLA: Tag/Post/Article ≤3 sec (US-1, US-2, US-5), Global ≤1 sec (US-3). Note: afterChange hook must use the saved document's actual slug — `revalidatePath(\`/vacancies/${doc.slug}\`, 'page')` — not the Next.js route template string.

### Decision 4: Sync script calls Payload REST API over HTTP

**Decision:** `scripts/sync-telegram.ts` on red server replaces Prisma with HTTP calls to `https://d-pub.ru/api/posts` (Payload REST endpoint). Pre-loads tag slug→id map via `GET /api/tags?limit=200` once per sync run.  
**Rationale:** Sync runs on red server (144.31.204.181), separate from the NetAngels process — Local API unavailable. REST API is the clean integration layer. Supports user-spec "sync stays on red server, only ORM changes". `setImmediate` on Passenger avoided.  
**Alternatives considered:** Direct `pg` Pool writes to Payload tables — bypasses Payload validation and hooks, brittle against Payload schema changes.

### Decision 5: Dedup via HTTP 409 on unique constraint

**Decision:** Sync script attempts `POST /api/posts` directly. On 409 Conflict response (Payload propagates PostgreSQL unique violation on `telegramMessageId + channelUsername`), the post is silently skipped.  
**Rationale:** Eliminates the current two-step "findUnique then create" pattern. Payload enforces the unique constraint at DB level regardless. [TECHNICAL]

### Decision 6: Agent API uses Payload API Keys on Users collection

**Decision:** `useAPIKey: true` on the `users` collection. Agent gets a dedicated User record with role `agent`. Access control on `articles` collection: `create` and `read` allowed for `agent` role; `update`, `delete`, `admin` operations restricted to `admin` role.  
**Rationale:** Token hashed, stored in DB, revocable via admin UI. Role separation: leaked agent token cannot modify Tags, Globals, or Posts. Supports user-spec security requirement.  
**Alternatives considered:** Custom middleware with hardcoded secret — no revocation, no audit trail.

### Decision 7: payload migrate via SSH in CI before touch reload

**Decision:** `deploy.yml` SSH step runs `cd ~/d-pub.ru/app && npx payload migrate` before `touch ~/d-pub.ru/reload`. If migration fails → no reload → old version keeps running.  
**Rationale:** Atomic deployment order: schema first, then traffic. Payload CLI is available via `node_modules/.bin/payload` — already deployed to server via `npm ci` step (to be added). `autoMigrate: true` risks serving traffic during migration. Supports user-spec deploy decision.

### Decision 8: Articles v1 — MDX + Payload hybrid

**Decision:** `/articles/page.tsx` merges MDX articles (from `lib/articles.ts`) and Payload articles (from Local API), sorted by `publishedAt` descending. `/articles/[slug]/page.tsx` tries Payload first; if not found, falls back to MDX.  
**Rationale:** MDX articles are established SEO content with working URLs — migrating them mid-project risks slug continuity. Agent-published articles need a home. Merger preserves both. Supports user-spec "статьи остаются в MDX в v1".  
**Alternatives considered:** Separate `/articles/new/[slug]` for Payload articles — creates two URL namespaces, bad for SEO consolidation.

### Decision 9: deploy.yml must also rsync source files needed for payload migrate

**Decision:** Add rsync steps for `app/`, `payload.config.ts`, `payload-migrations/` to deploy.yml — `npx payload migrate` on the server needs the compiled migration files.  
**Rationale:** Current deploy only rsyncs `.next/`, `lib/`, `generated/prisma/`, `public/`. Payload migrate reads migration JS files from `payload-migrations/` directory which doesn't exist on server yet. [TECHNICAL]  
**Alternatives considered:** Bundle migration files into `.next/` via webpack — not supported by Payload; migrations must be in their own directory.

### Decision 10: CSP updated only for /admin/\* via source matcher

**Decision:** In `next.config.mjs` `headers()`, add a second entry matching `source: '/admin(.*)'` with loosened `frame-src 'self'` (for Payload live preview iframes) and explicit `worker-src blob: 'self'` (for Lexical editor web workers). Global CSP unchanged.  
**Rationale:** Payload admin uses Lexical which spawns web workers via `blob:` URLs. Current global CSP lacks `worker-src` — admin UI would break without it. Scoping to `/admin/*` keeps public pages' CSP strict. Supports user-spec CSP risk mitigation.

## Data Models

### Payload Collections

**`tags` collection** (replaces Prisma `Tag`):

```typescript
{
  slug: 'tags',
  access: {
    read: () => true,
    create: ({ req }) => req.user?.role === 'admin',
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
  fields: [
    { name: 'name', type: 'text', required: true, unique: true },
    { name: 'slug', type: 'text', required: true, unique: true },
    { name: 'tagType', type: 'select', options: ['format', 'level', 'specialization'], required: true },
    { name: 'h1', type: 'text' },
    { name: 'seoTitle', type: 'text' },
    { name: 'seoDescription', type: 'textarea' },
    { name: 'seoText', type: 'richText' },  // Lexical
  ],
}
```

**`posts` collection** (replaces Prisma `Post` + `PostTag`):

```typescript
{
  slug: 'posts',
  access: {
    read: () => true,
    create: ({ req }) => req.user?.role === 'admin' || req.user?.role === 'sync',
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
  fields: [
    { name: 'type', type: 'select', options: ['vacancy', 'resume'], required: true },
    { name: 'title', type: 'text', required: true },
    { name: 'slug', type: 'text', unique: true },
    { name: 'description', type: 'textarea' },
    { name: 'company', type: 'text' },
    { name: 'salary', type: 'text' },
    { name: 'status', type: 'select', options: ['pending', 'published', 'rejected'], defaultValue: 'published' },
    { name: 'source', type: 'select', options: ['telegram', 'user'], defaultValue: 'telegram' },
    { name: 'telegramMessageId', type: 'text' },
    { name: 'channelUsername', type: 'text' },
    { name: 'imageUrl', type: 'text', validate: (val) => !val || /^https:\/\/.+/.test(val) || 'imageUrl must be HTTPS' },
    { name: 'tags', type: 'relationship', relationTo: 'tags', hasMany: true },
  ],
  // Composite unique index via custom SQL migration (Payload indexes config lacks partial-index syntax):
  // CREATE UNIQUE INDEX ON posts (telegram_message_id, channel_username)
  // WHERE telegram_message_id IS NOT NULL AND channel_username IS NOT NULL;
}
```

**`articles` collection** (new, for agent-published content):

```typescript
{
  slug: 'articles',
  fields: [
    { name: 'title', type: 'text', required: true },
    { name: 'slug', type: 'text', required: true, unique: true },
    { name: 'description', type: 'textarea' },
    { name: 'metaTitle', type: 'text' },
    { name: 'metaDescription', type: 'textarea' },
    { name: 'publishedAt', type: 'date' },
    { name: 'content', type: 'richText' },  // Lexical
    { name: 'status', type: 'select', options: ['draft', 'published'], defaultValue: 'draft' },
  ],
  access: {
    create: ({ req }) => req.user?.role === 'admin' || req.user?.role === 'agent',
    read: ({ req }) => req.user?.role === 'admin' ? true : { status: { equals: 'published' } },
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
}
```

**`media` collection** (Payload built-in, configured for local storage):

```typescript
{
  slug: 'media',
  upload: { staticDir: 'public/uploads', staticURL: '/uploads' },
  access: {
    read: () => true,
    create: ({ req }) => req.user?.role === 'admin',
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
}
```

**`users` collection** (Payload auth + API Keys):

```typescript
{
  slug: 'users',
  auth: { useAPIKey: true },
  access: {
    create: () => false,  // users created only via admin seed or admin UI; no self-registration
    read: ({ req }) => req.user?.role === 'admin',
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
  fields: [
    {
      name: 'role',
      type: 'select',
      options: ['admin', 'agent', 'sync'],
      defaultValue: 'agent',  // safe default: no admin privileges on creation
      access: {
        update: ({ req }) => req.user?.role === 'admin',  // only admin can change role
      },
    },
  ],
}
// Roles: 'admin' — full UI + API; 'agent' — articles create/read; 'sync' — posts create
```
