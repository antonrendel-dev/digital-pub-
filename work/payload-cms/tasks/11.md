## Details

**Files:**

- `scripts/sync-telegram.ts` — переписать (убрать Prisma, добавить Payload REST API)
- `tests/unit/sync-telegram.test.ts` — написать новый (или дополнить существующий)

**Dependencies:**

- Task 5 (Posts Collection) должен быть выполнен — Posts collection в Payload определена со всеми полями включая `telegramMessageId`+`channelUsername` unique constraint
- Task 8 (Replace Prisma Data Layer) должен быть выполнен — Prisma слой заменён на Payload
- Task 3 (Payload Installation) — users collection с полем role (agent/admin/sync) и useAPIKey: true настроены (через Task 5 transitive dependency)
- Users collection + sync role — созданы в Task 3 (Payload base config, onInit bootstrap)

**Edge cases:**

- 409 — это **не ошибка**. Это нормальный деdup-ответ от Payload (уникальный constraint на `telegramMessageId+channelUsername`). Логировать не нужно, просто вернуть `false`.
- `PAYLOAD_API_KEY` не должен попадать ни в один `console.log`/`console.error` ни напрямую, ни косвенно через объект headers или request config. Патерн: логировать только `res.status` и hardcoded URL-строку.
- `loadTagMap()` — запрос без авторизации (публичный read для Tags). Если вернулось 0 тегов — это не ошибка, просто посты сохранятся без тегов.
- `PAYLOAD_BASE_URL` — без trailing slash. Все fetch-запросы строятся как `${PAYLOAD_BASE_URL}/api/posts`.
- `backfillTags` и `backfillImages` удаляются полностью — не просто комментируются. Это уменьшает scope скрипта и убирает единственную причину, по которой скрипту нужен был Prisma для read-запросов.
- Поле `tags` в теле POST-запроса — массив строк-ID (Payload relationship format), не массив объектов и не массив slug-ов.

**Implementation hints:**

- `PostType` после удаления Prisma — определить локально как `type PostType = 'vacancy' | 'resume'`. Используется в интерфейсе `TelegramPost` и массиве `CHANNELS`.
- Для тестируемости — экспортировать `savePost` и `resolveTagIds` (или хотя бы `savePost` для TDD).
- В тестах мокать `global.fetch` через `jest.fn()`. `PAYLOAD_API_KEY` в тестах задавать через `process.env.PAYLOAD_API_KEY = 'test-api-key-secret'` в `beforeAll`.
- `sanitizeError` — оставить без изменений, он нужен для Telegram Bot API URL-ов (скрывает `bot<TOKEN>`). Для Payload API ошибок — не использовать `sanitizeError` (там нет секрета в URL), а логировать напрямую `res.status`.
