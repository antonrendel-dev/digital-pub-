## Description

Заменяем Prisma ORM на Payload Local API во всех файлах доступа к данным.

После того как Waves 1–3 создали Payload-конфиг, определили коллекции `tags` и `posts` и запустили миграцию данных, нам нужно переключить бизнес-логику с `prisma.*` на `getPayload()` + `payload.find()` / `payload.findByID()`. Это последний шаг, после которого в кодовой базе не останется ни одного импорта из `@prisma/*`.

Важно сохранить:

- существующие сигнатуры всех экспортируемых функций (они вызываются из app/ routes и компонентов)
- тип `FeedPost` из `lib/postUtils.ts` — он чисто-типовой файл без ORM-зависимостей
- паттерн `try/catch` + `console.warn('[posts] DB unavailable')` — он обеспечивает graceful degradation при недоступной БД
- функцию `toFeedPost` — нужно адаптировать под Payload-ответ (поле `tags` у поста — массив объектов `{id, name, slug, tagType}`, а не join-таблица `{ tag: { ... } }`)

После перезаписи удаляем `lib/prisma.ts` и `prisma/seed.ts` — они мертвы после миграции.

**Payload API reference:**

- `getPayload({ config: configPromise })` — импорт из `'payload'`, `configPromise` из `'@payload-config'`; возвращает singleton Payload instance
- `payload.find({ collection, where, limit, page, sort })` — возвращает `{ docs, totalDocs, totalPages, hasNextPage, hasPrevPage }`
- `payload.findByID({ collection, id })` — возвращает документ; при not found выбрасывает `NotFoundError` из `'payload/errors'` (не возвращает null). Для graceful handling оборачивать в `try/catch (e) { if (e instanceof NotFoundError) return null; throw e; }`
- Payload where-операторы: `{ field: { equals: value } }`, `{ field: { not_equals: null } }`, `{ field: { in: [id1, id2] } }`
- Сортировка: `sort: '-createdAt'` (минус = DESC)
- `limit: 0` — возвращает только `totalDocs` без документов (для count-операций)

**Edge cases:**

- `getTagsWithCounts()`: Payload не имеет нативного `_count` — нужно либо делать отдельный `payload.find` на posts для каждого тега (N+1, приемлемо если тегов < 100), либо denormalize count через afterChange hook на Posts (выходит за рамки этой задачи — использовать N+1 подход)
- `getPostById(id)`: Prisma использовал числовой `id`, Payload — тоже числовой (Drizzle adapter). Тип `id` сохраняется.
- `toFeedPost`: в Payload relationship поле `tags` с `hasMany: true` populate по умолчанию — возвращает массив полных объектов тегов. Убедиться что `depth` не ограничен (default depth=2 в Payload достаточно).
- `app/sitemap.ts` `updatedAt`: Payload Drizzle adapter хранит `updatedAt` как timestamp, возвращает Date — совместимо с `lastModified` в MetadataRoute.Sitemap.
- `prisma/seed.ts`: просто удалить, не проверять зависимостей — файл инициализировал тестовые данные через Prisma, после миграции не нужен.
- `getStats()` / `companyCount`: Payload Local API не имеет `groupBy`. Получить все посты с `depth: 0` и `limit: 0` не даст `docs` — нужно использовать достаточный `limit` (например, 10000) с `select: ['company']` если поддерживается, иначе получить все посты (приемлемо если записей < 50k) и дедуплицировать `company` в памяти через `new Set(docs.map(p => p.company).filter(Boolean)).size`.

**Implementation hints:**

- `getPayload` вызывается один раз в начале каждой функции: `const payload = await getPayload({ config: configPromise })`
- `payload.findByID()` в Payload v3 бросает `NotFoundError` (не возвращает null). Импортировать: `import { NotFoundError } from 'payload/errors'`. Использовать `payload.find()` + `docs[0] ?? null` для случаев where not-found должен возвращать null без исключения.
- Интерфейс `TagDetail` в `lib/tags.ts` должен включать `h1?: string | null` — маппить из Payload-документа тега (`tag.h1 ?? null`).
- Для `getTagsWithCounts` — проще всего сначала получить все посты с тегами через `payload.find({ collection: 'posts', where: { status: { equals: 'published' } }, limit: 0, depth: 1 })`, затем агрегировать count в памяти по tag.id — избегает N+1 запросов
- В `toFeedPost` поле `createdAt` у Payload — Date объект (Drizzle), нужно `.toISOString()`
- Импорт config: `import config from '@payload-config'` (alias настроен в Task 3 в `next.config.mjs` через `withPayload()`)
