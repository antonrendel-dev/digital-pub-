# Task 16: Deploy

## Required Skills

Перед выполнением задачи загрузи:

## Description

Производственный деплой Payload CMS с безопасной процедурой переключения (cutover). Задача выполняется только после того, как Task 15 (Pre-deploy QA на staging) подтвердила полную работоспособность всех сценариев.

Порядок действий строго последовательный: сначала отключаем sync (чтобы не было новых данных во время миграции), делаем backup базы, деплоим код через CI, запускаем `payload migrate` на сервере (SSH, автоматически из `deploy.yml`), проверяем дымом, включаем sync обратно. Если `payload migrate` падает — `touch reload` не выполняется, старая версия продолжает работать, восстанавливаем из дампа.

Ожидаемый даунтайм: ~5–10 минут (время `payload migrate` + перезапуск процесса).

### 2. Деплой через GitHub Actions

**Шаг 4 — Смержить dev → main и запустить CI:**

```bash
# Локально в репозитории
git checkout main
git merge dev --no-ff -m "feat: merge Payload CMS integration to main"
git push origin main
```

GitHub Actions автоматически запустит `deploy.yml` при push в `main`.

Следить за ходом деплоя в GitHub Actions UI (или через gh CLI):

```bash
gh run list --workflow=deploy.yml --limit=3
gh run watch   # реального времени лог последнего run
```

**Шаг 5 — Что делает deploy.yml (контролируем каждый шаг):**

Согласно `.github/workflows/deploy.yml`, CI выполняет следующие шаги:

1. `actions/checkout@v4`
2. `actions/setup-node@v4` — Node.js 22, кеш npm
3. `npm ci` — установка зависимостей
4. `npx prisma generate` (старый шаг; после Task 2 должен быть заменён или убран)
5. `npm run build` — сборка Next.js
6. `rsync .next/` → NetAngels
7. Дополнительные rsync шагов (Prisma client, lib/, public/)
8. **SSH: `npx payload migrate`** — применяет Payload миграции на prod БД
9. **Если `payload migrate` exit 0 → SSH: `touch ~/d-pub.ru/reload`** — перезапуск приложения

Если на шаге 8 `payload migrate` возвращает ненулевой exit code — шаг `touch reload` (шаг 9) не выполняется. Старая версия продолжает работать.

**Критически важно:** Убедиться в GitHub Actions логах что шаги выполнились именно в таком порядке: migrate → reload.

## Rollback Procedure

Если `payload migrate` завершился с ошибкой (GitHub Actions показывает fail на SSH migrate шаге):

1. `touch reload` НЕ выполнился → старая версия продолжает работать → сайт доступен.
2. Исправить причину ошибки миграции (обычно конфликт схемы или проблема с соединением к БД).
3. Если данные были повреждены — восстановить из дампа:

```bash
ssh c48127@91.201.52.231
# Найти свежий дамп
ls -lh ~/backups/dpub_pre_payload_*.sql
# Восстановить (ВНИМАНИЕ: перезапишет текущую БД)
psql "$DB_CONNECTION_STRING" < ~/backups/dpub_pre_payload_YYYYMMDD_HHMMSS.sql
```

4. После восстановления данных — не выполнять `touch reload` до устранения причины сбоя.
5. Включить sync обратно только после подтверждения что сайт работает корректно.

Если нужно полностью откатить код на предыдущий коммит:

```bash
ssh c48127@91.201.52.231
cd ~/d-pub.ru/app/
git checkout <предыдущий_коммит>
export NODE_ENV=development && npm run build
cd ~/d-pub.ru/ && touch reload
```

## Details

**Серверы:**

- App server: `c48127@91.201.52.231` — NetAngels, приложение в `~/d-pub.ru/`
- Red proxy server: `root@144.31.204.181` — cron sync, staging
- Prod URL: `https://d-pub.ru`

**Порядок операций критичен:**

1. Disable sync → 2. pg_dump → 3. migrate-tags.ts → 4. migrate-posts.ts → 5. merge + push → 6. CI: payload migrate → 7. CI: touch reload → 8. smoke verify → 9. re-enable sync

**`touch reload` — единственный механизм перезапуска** на NetAngels (нет pm2/systemd). Выполняется только если `payload migrate` вернул exit 0.

**Env vars на prod сервере:**

- `DB_CONNECTION_STRING` — PostgreSQL строка подключения (не `DATABASE_URL`)
- `PAYLOAD_SECRET` — минимум 32 символа
- `PAYLOAD_ADMIN_EMAIL`, `PAYLOAD_ADMIN_PASSWORD` — для onInit bootstrap
- `PAYLOAD_API_KEY` — API ключ sync-пользователя для красного сервера

**Backup location:** `~/backups/` на app-сервере (создать директорию если не существует: `mkdir -p ~/backups`).

**Даунтайм:** сайт не отвечает в период `payload migrate` → `touch reload` → Passenger restart (~5–10 минут). Заранее предупредить если нужно.

**Staging must pass Task 15 QA before production deploy** — эта задача не выполняется если Task 15 не завершена с положительным результатом.
