# Task 5: Posts Collection + Migration Script

## Required Skills

Перед выполнением задачи загрузи:

## Description

Заполняем стаб `payload/collections/posts.ts` полным определением коллекции Posts согласно Data Models в tech-spec. Коллекция заменяет Prisma-таблицы `Post` + `PostTag`.

Ключевые особенности:

- **Access control:** роль `sync` может создавать посты (для sync-скрипта на красном сервере через REST API), только `admin` — обновлять и удалять
- **afterChange hook:** вызывает `revalidatePath` для страниц вакансий (используем реальный slug документа, не шаблонный путь)
- **Post duplication:** кастомный admin list action (кнопка в UI Payload для дублирования поста)
- **Composite unique index:** частичный (PARTIAL) unique index — Payload `indexes` config не поддерживает `WHERE` clause, поэтому нужна кастомная SQL-миграция

Пишем `scripts/migrate-posts.ts` — одноразовый скрипт переноса данных из Prisma-таблиц (`post`, `post_tag`) в Payload-коллекцию Posts с сохранением всех связей тегов.

| `title`              | `title`                 |                                                                                                                                      |
| `slug`               | `slug`                  | nullable в Prisma → nullable в Payload                                                                                               |
| `description`        | `description`           |                                                                                                                                      |
| `company`            | `company`               |                                                                                                                                      |
| `salary`             | `salary`                |                                                                                                                                      |
| `status`             | `status`                | `pending`/`published`/`rejected`                                                                                                     |
| `source`             | `source`                | `telegram`/`user`                                                                                                                    |
| `imageUrl`           | `imageUrl`              | validate: HTTPS only                                                                                                                 |
| `telegramMessageId`  | `telegramMessageId`     |                                                                                                                                      |
| `channelUsername`    | `channelUsername`       |                                                                                                                                      |

| `createdAt`          | `createdAt`             | Передавать явно чтобы сохранить хронологический порядок ленты; Payload Drizzle-адаптер принимает это поле при `overrideAccess: true` |

**Edge cases:**

- Посты без тегов (PostTag записей нет) → `tags: []` в Payload, не ошибка
- Посты с `slug: null` → передавать `null`, Payload не требует slug
- Дублирование поста (admin action): новый пост должен получить пустые `telegramMessageId` и `channelUsername` (иначе нарушится PARTIAL unique index), а slug должен быть `null` или уникальным суффиксом
- `imageUrl` validate функция — проверяет `^https://` только если значение непустое (`!val || /^https:\/\/.+/.test(val)`)
- afterChange hook: `doc.slug` может быть `null` для постов без slug — проверять перед `revalidatePath`

**afterChange hook — точная стратегия revalidation:**

Согласно Decision 3 в tech-spec:

```typescript
// Post saved → revalidatePath для категории + общего листинга
afterChange: ({ doc }) => {
  if (doc.slug) {
    revalidatePath(`/vacancies/${doc.slug}`, 'page')
  }
  revalidatePath('/vacancies', 'layout')
}
```

**Admin list action "Дублировать пост":**

Payload Custom Admin Component (list action). При нажатии:

1. Берёт выбранный пост
2. Копирует все поля
3. Обнуляет: `slug`, `telegramMessageId`, `channelUsername` (чтобы не нарушить unique constraint)
4. Устанавливает `status: 'pending'` (безопасный дефолт для дубликата)
5. Создаёт новый документ через `payload.create({ ..., overrideAccess: true })`

**Partial index vs Payload indexes config:**

Payload `indexes` в field config поддерживает `{ unique: true }`, но не поддерживает `WHERE` clause. Поэтому:

- В поле `telegramMessageId` и `channelUsername` — НЕ добавлять `unique: true` (это создаст обычный unique index, который не работает с NULL значениями корректно)
- Использовать `payload-migrations/custom-posts-unique-idx.sql` для PARTIAL unique index
- SQL-миграция выполняется вручную или через CI как `psql < payload-migrations/custom-posts-unique-idx.sql` после `payload migrate`
