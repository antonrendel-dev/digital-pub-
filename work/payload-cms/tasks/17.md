## Description

Финальная верификация продакшена после деплоя Payload CMS. Агент проверяет все smoke-маршруты через curl, удостоверяется что `/admin` редиректит на логин, тестирует API-авторизацию (с токеном → 201, без токена → 401), и запускает sync-скрипт на красном сервере через SSH — после чего убеждается что количество постов в Payload выросло.

Revalidation SLA (Tag edit → страница обновляется ≤3 сек; Navbar edit → ≤1 сек) требует браузера для замера задержки — это **пользовательская проверка**, не агентская.

Верификация производится на **production** (https://d-pub.ru), а не на staging.

## Verification Steps

### Smoke (агент выполняет curl)

```bash
# 1. Главная
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/
# → 200

# 2. Список вакансий
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/vacancies
# → 200

# 3. Страница тега SMM — должна содержать seoText
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/vacancies/smm
# → 200
curl -s https://d-pub.ru/vacancies/smm | grep -c "seoText\|<p\|<section"
# → ≥1 (убедиться что seoText рендерится в разметке)

# 4. Статьи
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/articles
# → 200

# 5. Резюме
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/resumes
# → 200

# 6. Admin — редирект на логин (302)
curl -s -o /dev/null -w '%{http_code}' -L https://d-pub.ru/admin
# Ожидается редирект на /admin/login; без -L проверить Location header:
curl -sI https://d-pub.ru/admin | grep -i "location\|HTTP/"
# → HTTP/... 302 и Location: .../admin/login

# 7. POST /api/articles без токена → 401
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://d-pub.ru/api/articles \
  -H "Content-Type: application/json" \
  -d '{"title":"Verification test","slug":"verify-test","status":"draft"}'
# → 401

# 8. POST /api/articles с agent API Key → 201
# PAYLOAD_API_KEY берётся из переменной окружения на сервере или из секретов
curl -s -o /dev/null -w "%{http_code}" \
  -X POST https://d-pub.ru/api/articles \
  -H "Authorization: users API-Key ${PAYLOAD_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"title":"Verification test","slug":"verify-test-agent","status":"draft"}'
# → 201

# 9. Sitemap
curl -s -o /dev/null -w "%{http_code}" https://d-pub.ru/sitemap.xml
# → 200
```

### Sync verification (SSH на NetAngels)

```bash
# Перед запуском — получить текущий count постов через Payload REST API
BEFORE=$(curl -s "https://d-pub.ru/api/posts?limit=0" | grep -o '"totalDocs":[0-9]*' | grep -o '[0-9]*')
echo "Posts before sync: $BEFORE"

# Запустить sync-скрипт на NetAngels сервере
ssh c48127@91.201.52.231 "bash /home/c48127/d-pub.ru/app/scripts/auto-sync.sh"

# После запуска — проверить рост
AFTER=$(curl -s "https://d-pub.ru/api/posts?limit=0" | grep -o '"totalDocs":[0-9]*' | grep -o '[0-9]*')
echo "Posts after sync: $AFTER"

# Убедиться что AFTER > BEFORE (или равно, если нет новых постов в каналах — это нормально)
# Главное — скрипт завершился без ошибок (exit 0)
```

## Details

**Серверная топология:**

- Production app: `c48127@91.201.52.231` (NetAngels), порт 3001, приложение в `~/d-pub.ru/`
- Red server (proxy + sync cron): `root@144.31.204.181`
- Sync script на red server: `/opt/bots/telegram-bot-vac/scripts/sync-telegram.ts`

**PAYLOAD_API_KEY:**

- Это API Key пользователя с ролью `sync`, созданного в Task 3 (`onInit` hook)
- Значение хранится в env на NetAngels сервере — прочитать через `ssh c48127@91.201.52.231 "grep PAYLOAD_API_KEY ~/d-pub.ru/app/.env"`
- В curl передавать как: `Authorization: users API-Key <значение>`
- Никогда не логировать значение ключа в отчётах или решениях

**Проверка seoText на /vacancies/smm:**

- seoText — это Lexical-контент, рендерится как HTML в страницу
- Достаточно убедиться что страница возвращает 200 и содержит контент (не пустую страницу)
- Если после Task 8 тег SMM содержит seoText в Payload — он должен рендериться в разметке

**Повторный sync (dedup):**

- По желанию: запустить sync дважды и убедиться что count не изменился на второй итерации (409 dedup работает)
- Это не обязательная проверка, но хорошая дополнительная валидация Decision 5

**Если POST /api/articles → 201 создал реальную статью:**

- Удалить тестовую статью через Payload admin после верификации (чтобы не засорять продакшн)
