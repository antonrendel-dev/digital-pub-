# Task 4: Tags Collection + Migration Script

## Required Skills

Перед выполнением задачи загрузи:

- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Задача 3 создала стаб `payload/collections/tags.ts` — пустую заглушку. Здесь мы наполняем его полным определением коллекции Tags согласно Data Models из tech-spec: поля, access control, afterChange hook с `revalidatePath`.

Параллельно пишем одноразовый миграционный скрипт `scripts/migrate-tags.ts`, который переносит все теги из Prisma-таблицы `tag` (включая SEO-поля) в Payload через Local API. Поле `h1` новое — в Prisma его нет, в миграции оно выставляется `null`. Поле `seoText` в Prisma — обычная строка; при переносе оборачивается в минимальную Lexical JSON структуру.

Скрипт предназначен для однократного запуска на staging, затем на проде при cutover.

## What to do

1. Заполнить `payload/collections/tags.ts` полным определением коллекции (вместо стаба из Task 3).
2. Реализовать access control: `read: () => true`, `create/update/delete: ({ req }) => req.user?.role === 'admin'`.
3. Добавить `afterChange` хук: вызывает `revalidatePath(\`/vacancies/${doc.slug}\`, 'page')`и`revalidatePath('/', 'layout')`. Импортировать `revalidatePath`из`'next/cache'`.
4. Написать `scripts/migrate-tags.ts`:
   - Подключиться к Prisma (через существующий `lib/prisma.ts`) для чтения тегов из БД.
   - Инициализировать Payload через `getPayload({ config })`.
   - Для каждого тега из Prisma: вызвать `payload.create({ collection: 'tags', data: { ... } })`.
   - Поле `h1` — не существует в Prisma, передавать `null` (или не передавать — Payload проставит `null` по умолчанию).
   - Поле `seoText` — если в Prisma не `null`, обернуть в Lexical JSON структуру (см. Details). Если `null` — передать `null`.
   - После миграции: проверить что `count(prisma.tag)` === `count(payload.find({ collection: 'tags' }).totalDocs)`.
5. Написать unit-тесты (TDD — до реализации).

## Details

**Files:**

- `payload/collections/tags.ts` — заполнить стаб полным определением коллекции
- `scripts/migrate-tags.ts` — создать с нуля миграционный скрипт

**Lexical JSON wrapper для `seoText`:**

```typescript
function toRichText(plainText: string | null) {
  if (!plainText) return null
  return {
    root: {
      type: 'root',
      version: 1,
      children: [
        {
          type: 'paragraph',
          version: 1,
          children: [
            {
              type: 'text',
              version: 1,
              text: plainText,
              format: 0,
            },
          ],
        },
      ],
    },
  }
}
```

**Структура tags collection (из tech-spec Data Models):**

```typescript
{
  slug: 'tags',
  access: {
    read: () => true,
    create: ({ req }) => req.user?.role === 'admin',
    update: ({ req }) => req.user?.role === 'admin',
    delete: ({ req }) => req.user?.role === 'admin',
  },
  fields: [
    { name: 'name', type: 'text', required: true, unique: true },
    { name: 'slug', type: 'text', required: true, unique: true },
    { name: 'tagType', type: 'select', options: ['format', 'level', 'specialization'], required: true },
    { name: 'h1', type: 'text' },
    { name: 'seoTitle', type: 'text' },
    { name: 'seoDescription', type: 'textarea' },
    { name: 'seoText', type: 'richText' },
  ],
  hooks: {
    afterChange: [
      async ({ doc }) => {
        revalidatePath(`/vacancies/${doc.slug}`, 'page')
        revalidatePath('/', 'layout')
      },
    ],
  },
}
```

**afterChange hook — важное:**

- Импорт: `import { revalidatePath } from 'next/cache'`
- Использовать `doc.slug` (фактический slug сохранённого тега), НЕ шаблонную строку `'/vacancies/[category]'`
- Decision 3 (tech-spec): "afterChange hook must use the saved document's actual slug"

**migrate-tags.ts — структура скрипта:**

- Импортировать Payload config и `getPayload` из `payload`
- Читать теги из Prisma через `prisma.tag.findMany()`
- Для каждого тега вызвать `payload.create({ collection: 'tags', data: { ... }, overrideAccess: true })`
- `overrideAccess: true` нужен — скрипт запускается не от имени admin-пользователя
- Count assertion в конце: сравнить `prismaCount` с `(await payload.find({ collection: 'tags', limit: 0 })).totalDocs`
- Логировать прогресс (`console.log`) и ошибки (`console.error`) по каждому тегу
- Завершить процесс: `process.exit(0)` или `process.exit(1)` по итогу

**Prisma Tag model (из schema.prisma):**

```
id              Int
name            String   (unique)
slug            String   (unique)
tagType         TagType  (format | level | specialization)
seoTitle        String?
seoDescription  String?
seoText         String?
```

Поле `h1` — отсутствует в Prisma. В migrate-tags.ts передавать как `h1: null`.

**Dependencies:**

- Task 3 должна быть выполнена: `payload/collections/tags.ts` стаб существует, Payload установлен и настроен
- `payload.config.ts` должен импортировать Tags collection

**Edge cases:**

- Тег с `seoText = null` — `toRichText(null)` возвращает `null`, не пустой объект
- Тег с `seoText = ''` (пустая строка) — трактовать как `null` (falsy check)
- При дублировании slug (повторный запуск скрипта) — Payload выбросит ошибку уникальности; добавить try/catch и логировать как `skip (already exists)`, не падать полностью
- Большое количество тегов — sequential processing достаточен (тегов ~17, не сотни тысяч)

**Implementation hints:**

- `getPayload` импортируется из `payload`, config — из `../../payload.config` (относительный путь от `scripts/`)
- При запуске через `tsx` (не `ts-node`) Payload инициализируется корректно в Node.js контексте без Next.js сервера
- `options: ['format', 'level', 'specialization']` для поля `tagType` — в Payload select это массив строк или объектов `{ label, value }`; оба варианта допустимы, предпочесть простой массив строк для консистентности
