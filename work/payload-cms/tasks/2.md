## Description

Обновляем `deploy.yml` и создаём `deploy-staging.yml` для деплоя с Payload CMS. Обе задачи параллельны с Task 1 (Next.js 15 upgrade), зависимостей нет.

Текущий `deploy.yml` содержит шаги `Generate Prisma client` и `Deploy Prisma client` — их надо убрать. Вместо этого:

- `public/uploads/` должен быть исключён из rsync (иначе загруженные медиафайлы сотрутся при каждом деплое)
- `payload-migrations/` нужно деплоить на сервер (необходим для `npx payload migrate`)
- SSH-шаг `npx payload migrate` должен выполняться ДО `touch reload` — атомарный порядок: если миграция упала, приложение не перезапускается и старая версия продолжает работать

Staging (`deploy-staging.yml`) — новый файл, его не существует. Деплоит из ветки `dev` на красный сервер (144.31.204.181, порт 3002). Использует отдельные GitHub Secrets, никогда не шарит значения с продом.

**Staging server:** `claude@144.31.204.181`, app path `~/staging.d-pub.ru/app/`, reload: `touch ~/staging.d-pub.ru/reload`

**Atomic order — критично:**
GitHub Actions останавливает workflow при ненулевом exit code шага. Два отдельных шага в правильном порядке гарантируют атомарность автоматически:

```
Step: Run payload migrate   ← если упало → workflow стоп, следующий шаг не запускается
Step: Restart app           ← touch reload — выполняется только если migrate прошла успешно
```

**`--ignore-missing-args` для payload-migrations/:**
Директория `payload-migrations/` появится только после первой committed миграции Payload. До этого момента rsync без `--ignore-missing-args` упадёт с ошибкой "source directory not found". Флаг позволяет шагу пройти успешно даже если директории нет.
