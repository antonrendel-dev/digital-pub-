# Task 13: Security Audit

## Required Skills

Перед выполнением задачи загрузи:

- `/skill:security-auditor` — [skills/security-auditor/SKILL.md](~/.claude/skills/security-auditor/SKILL.md)

## Description

Аудит безопасности всего кода, созданного или изменённого в рамках фичи Payload CMS. Задача выполняется параллельно с Tasks 12 и 14 (Audit Wave) — после завершения Wave 5 (Tasks 10 и 11).

Auditor IS the review — внешних ревьюеров нет. Результаты фиксируются в `decisions.md` с severity-рейтингом. P0-находки блокируют деплой.

Фокус проверки:

1. **`/api/articles` route** — правильность аутентификации через `payload.auth({ headers })`: может ли токен от одной коллекции пройти аутентификацию для другой? Убедиться что `overrideAccess: true` отсутствует.
2. **Articles access control** — агент может создавать статьи, но не может их обновлять/удалять; публичный `read` возвращает только опубликованные.
3. **Posts access control** — только `sync`-роль создаёт посты, только `admin` может обновлять/удалять.
4. **Tags/Media access control** — только `admin` пишет.
5. **Users collection** — самостоятельная регистрация невозможна (`create: () => false`), `role` имеет `defaultValue: 'agent'`.
6. **PAYLOAD_SECRET** — задокументирована ли минимальная энтропия в `.env.example`?
7. **PAYLOAD_API_KEY** — не утекает ли значение API-ключа в логи? Проверить паттерн `sanitizeError` в sync-скрипте — заголовок `Authorization` не должен попасть в `console.error`.
8. **CSP для `/admin(.*)`** — включает ли заголовок `worker-src blob: 'self'`? Нет ли `unsafe-eval` в scope публичных страниц?
9. **imageUrl validation** — присутствует ли field-level `validate` функция (HTTPS only) в коллекции `posts`?
10. **middleware.ts** — корректно ли защищён маршрут `/admin`? Нет ли bypass через trailing slash или query params?

## What to do

1. **Прочитай** tech-spec и user-spec — понять архитектурные решения по безопасности (Decision 6, Decision 10).
2. **Прочитай** все файлы коллекций и глобалов: `payload/collections/*.ts`, `payload/globals/*.ts`.
3. **Прочитай** `app/api/articles/route.ts` — проверить полный auth flow.
4. **Прочитай** `scripts/sync-telegram.ts` — проверить логирование ошибок.
5. **Прочитай** `next.config.mjs` — проверить CSP-заголовки для `/admin(.*)`.
6. **Прочитай** `middleware.ts` — проверить защиту `/admin/*`.
7. **Прочитай** `.env.example` — проверить документирование PAYLOAD_SECRET.
8. **Для каждой находки** определить severity:
   - **P0 (Critical)** — эксплуатируется без аутентификации, ведёт к RCE, утечке секретов или полному обходу AC → блокирует деплой
   - **P1 (High)** — требует аутентификации, но позволяет превысить привилегии (privilege escalation)
   - **P2 (Medium)** — защитная мера отсутствует, риск ограничен контекстом
   - **P3 (Low/Info)** — best-practice отклонение, не эксплуатируемо в текущей конфигурации
9. **Запиши отчёт** в `decisions.md` по шаблону ниже.

## Security Checklist

### /api/articles route

- [ ] `payload.auth({ headers })` вызывается с оригинальными `request.headers` — не с подделанными
- [ ] `overrideAccess: true` отсутствует в `payload.create({ collection: 'articles', ... })`
- [ ] Токен типа `users API-Key <token>` для пользователя с ролью `sync` не проходит через проверку `role === 'agent'` (cross-collection token misuse)
- [ ] Токен типа `users API-Key <token>` для пользователя без роли (`agent` default) не позволяет `update`/`delete`
- [ ] 401 возвращается при отсутствии заголовка `Authorization`
- [ ] 401 возвращается при невалидном токене
- [ ] 400 возвращается при валидном токене но невалидном body (отсутствует `title`)

### Articles collection access control

- [ ] `create` разрешён для ролей `admin` и `agent`
- [ ] `read` для неаутентифицированных возвращает только `{ status: { equals: 'published' } }`
- [ ] `update` разрешён только для `admin`
- [ ] `delete` разрешён только для `admin`
- [ ] Агент с валидным API-ключом НЕ может вызвать `update` через Payload REST API напрямую

### Posts collection access control

- [ ] `create` разрешён для `admin` и `sync` (не для `agent`)
- [ ] `update` разрешён только для `admin`
- [ ] `delete` разрешён только для `admin`
- [ ] Публичный `read` возвращает все посты (посты публичные — это ожидаемо)

### Tags collection access control

- [ ] `create`, `update`, `delete` разрешены только для `admin`
- [ ] `read` открыт публично (`() => true`)

### Media collection access control

- [ ] `create`, `update`, `delete` разрешены только для `admin`
- [ ] `read` открыт публично

### Users collection

- [ ] `create: () => false` — самостоятельная регистрация невозможна через REST API
- [ ] `role` field `defaultValue: 'agent'` — безопасный дефолт
- [ ] `role` field `access.update` ограничен только для `admin` — нельзя сменить себе роль
- [ ] `read` ограничен только `admin` — список пользователей не утекает

### Globals access control

- [ ] Все Globals (`navbar`, `footer`, `social-channels`, `site-settings`) — `update` только для `admin`

### PAYLOAD_SECRET

- [ ] В `.env.example` указан минимальный размер (32+ символа) и пример генерации (`openssl rand -hex 32`)
- [ ] Значение не захардкожено нигде в коде (`grep -r "PAYLOAD_SECRET" --include="*.ts" --include="*.mjs"` — только env-ссылки)

### Sync script (PAYLOAD_API_KEY)

- [ ] Заголовок `Authorization: users API-Key <value>` не попадает в `console.error` / `console.log`
- [ ] В блоке `catch` логируется только `response.status` и URL без заголовков запроса
- [ ] Паттерн `sanitizeError` сохранён (Telegram BOT_TOKEN не утекает в логи)
- [ ] `PAYLOAD_API_KEY` не логируется даже при успешном ответе

### CSP для /admin

- [ ] В `next.config.mjs` `headers()` присутствует отдельный объект с `source: '/admin(.*)'`
- [ ] Заголовок включает `worker-src blob: 'self'` (для Lexical web workers)
- [ ] Заголовок включает `frame-src 'self'` (для Payload live preview)
- [ ] `unsafe-eval` отсутствует в CSP публичных страниц (source: '/(.\*)')
- [ ] `unsafe-inline` отсутствует в `script-src` публичных страниц

### middleware.ts

- [ ] Маршруты `/admin` и `/admin/*` защищены
- [ ] Нет обхода через `//admin` или `/admin/` (trailing slash)
- [ ] Публичные маршруты (`/api/articles` GET, `/api/tags`, `/api/posts`) доступны без авторизации

### imageUrl validation

- [ ] В `payload/collections/posts.ts` поле `imageUrl` содержит `validate` функцию
- [ ] Validate отклоняет `http://` URL (только HTTPS)
- [ ] Validate отклоняет `javascript:` URL

### Dependencies

- Depends on Tasks 10 and 11 (Wave 5) being complete — the files being audited must exist
- Runs in parallel with Task 12 (Code Audit) and Task 14 (Test Audit) — no shared state, no conflicts

### Edge cases

- **middleware.ts may not exist** — Task 3 description notes that Payload's built-in `/admin` redirect may be sufficient and middleware can be omitted. Before auditing `middleware.ts`, check if the file exists. If absent, note this explicitly in the report with the rationale from Task 3.
- **Cross-collection token misuse** — `payload.auth({ headers })` authenticates against the `users` collection. Verify that a token issued for a `sync` user (role) cannot pass the `role === 'agent'` guard in the route handler. This is a logic check in route code, not a Payload config issue.
- **`overrideAccess` defaults to false** — absence of the flag is correct; only flag it if `overrideAccess: true` is explicitly set.

OWASP checklist to apply: A01 Broken Access Control (collection AC rules), A02 Cryptographic Failures (PAYLOAD_SECRET entropy), A03 Injection (imageUrl validate), A05 Security Misconfiguration (CSP headers), A09 Security Logging and Monitoring Failures (API key in logs).
