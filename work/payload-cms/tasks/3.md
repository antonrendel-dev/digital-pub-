# Task 3: Payload Installation & Base Config

## Required Skills

Перед выполнением задачи загрузи:

- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Устанавливаем Payload CMS v3 пакеты и создаём основу конфигурации. Это фундаментальный шаг Phase 2 — после него Wave 3 задачи (Tasks 4–7) могут запускаться параллельно и наполнять стаб-коллекции.

Что создаём:

- `payload.config.ts` — корневой конфиг: импортирует стабы из `payload/collections/*` и `payload/globals/*`, db-postgres адаптер с явным `url: process.env.DB_CONNECTION_STRING`, richtext-lexical, onInit хук для начального сида admin и sync пользователей
- Стаб-файлы для всех коллекций и глобалов — пустые экспорты, которые Task 4–7 заполнят
- Payload v3 обрабатывает аутентификацию `/admin` нативно — кастомный `middleware.ts` не требуется
- Обновление `next.config.mjs` — `withPayload()` обёртка + scoped CSP для `/admin(.*)`
- Обновление `.env.example` — новые переменные для Payload

Ключевой нюанс: env var для БД называется `DB_CONNECTION_STRING` (не `DATABASE_URI`) — это NetAngels-соглашение. В db-postgres адаптере передаём его явно через `url: process.env.DB_CONNECTION_STRING`.

## What to do

1. Установить npm-пакеты: `payload`, `@payloadcms/next`, `@payloadcms/db-postgres`, `@payloadcms/richtext-lexical`, `sharp`

2. Создать директории `payload/collections/` и `payload/globals/`

3. Создать стаб-файлы коллекций (минимальный экспорт — только slug и пустые fields, Wave 3 заполнит):
   - `payload/collections/tags.ts`
   - `payload/collections/posts.ts`
   - `payload/collections/articles.ts`
   - `payload/collections/media.ts`
   - `payload/collections/users.ts`

4. Создать стаб-файлы глобалов:
   - `payload/globals/navbar.ts`
   - `payload/globals/footer.ts`
   - `payload/globals/socialChannels.ts`
   - `payload/globals/siteSettings.ts`

5. Создать `payload.config.ts`:
   - Импортировать все стабы
   - `db: postgresAdapter({ url: process.env.DB_CONNECTION_STRING })`
   - `editor: lexicalEditor({})`
   - `secret: process.env.PAYLOAD_SECRET`
   - `serverURL: 'https://d-pub.ru'` — захардкодить значение напрямую. Переменная `NEXT_PUBLIC_BASE_URL` в проекте отсутствует; если в будущем потребуется env var для base URL — добавить в `.env.example`.
   - `collections: [Tags, Posts, Articles, Media, Users]`
   - `globals: [Navbar, Footer, SocialChannels, SiteSettings]`
   - `onInit` хук: если нет ни одного User — создать admin из `PAYLOAD_ADMIN_EMAIL`/`PAYLOAD_ADMIN_PASSWORD` через `payload.create({ collection: 'users', ... })`, затем создать sync-пользователя с `role: 'sync'` и `enableAPIKey: true` (полученный `apiKey` логировать один раз с `console.log('[payload onInit] Sync API Key:', result.apiKey)`)

6. Обновить `next.config.mjs`:
   - Добавить импорт: `import { withPayload } from '@payloadcms/next/withPayload'`
   - Обернуть экспорт: `export default withPayload(nextConfig)`
   - Настроить заголовки Content-Security-Policy для маршрутов `/admin(.*)`, чтобы разрешить JavaScript Payload (включая `worker-src blob: 'self'`)
   - Заменить `X-Frame-Options: SAMEORIGIN` на `X-Frame-Options: DENY` в глобальных headers для публичных страниц (чтобы не конфликтовало с `frame-ancestors 'self'` на /admin)

7. **Не создавать `middleware.ts`** — Payload v3 обрабатывает аутентификацию и редиректы `/admin` нативно. Никакой кастомный middleware не нужен. Субпуть `@payloadcms/next/middleware` не существует в npm exports Payload v3.

8. Обновить `.env.example` — добавить блок:

   ```
   # Payload CMS
   PAYLOAD_SECRET=           # min 32 chars, generate: openssl rand -hex 32
                             # WARNING: rotating this value invalidates all sessions and API Keys
   PAYLOAD_ADMIN_EMAIL=
   PAYLOAD_ADMIN_PASSWORD=
   PAYLOAD_API_KEY=          # filled after first onInit run (check server logs)
   DB_CONNECTION_STRING=     # postgresql://user:password@host:5432/dbname
   ```

9. Убедиться что `npm run build` проходит (TypeScript + Next.js компиляция)

### Smoke

Запустить локально (`npm run dev` или `npm run build && npm start`), затем:

```bash
# 1. /admin редиректит на login
curl -sI http://localhost:3000/admin | grep -E "^HTTP|^location"
# ожидаем: HTTP/... 302, location: /admin/login

# 2. CSP на /admin/login содержит worker-src blob:
curl -sI http://localhost:3000/admin/login | grep Content-Security-Policy
# ожидаем: в строке есть "worker-src blob:"

# 3. Стаб-файлы существуют
ls payload/collections/ payload/globals/
# ожидаем: tags.ts posts.ts articles.ts media.ts users.ts (в collections)
#          navbar.ts footer.ts socialChannels.ts siteSettings.ts (в globals)

# 4. DB_CONNECTION_STRING нигде не aliased
grep -r "DATABASE_URI\|DATABASE_URL" payload.config.ts || echo "clean"
# ожидаем: "clean"
```

**Dependencies:**

- Новые npm пакеты: `payload`, `@payloadcms/next`, `@payloadcms/db-postgres`, `@payloadcms/richtext-lexical`, `sharp`

**Edge cases:**

- `PAYLOAD_SECRET` должен быть минимум 32 символа — Payload валидирует при старте и упадёт с ошибкой если короче. В `.env.example` обязательно указать команду: `openssl rand -hex 32`
- Ротация `PAYLOAD_SECRET` инвалидирует все сессии и API Keys — предупредить в комментарии `.env.example`
- onInit хук запускается при каждом старте процесса — обязательно проверить `count(users)` перед созданием, иначе при каждом рестарте будет создаваться новый sync-пользователь
- `withPayload()` в `next.config.mjs` — должна быть ESM-совместимая версия (файл `.mjs`). Импорт через `import { withPayload } from '@payloadcms/next/withPayload'` с `await` если конфиг async, без `await` если sync
- Текущий `next.config.mjs` использует `export default nextConfig` — нужно изменить на `export default withPayload(nextConfig)`. Если withPayload принимает async конфиг — привести к правильному формату по документации Payload v3
- CSP на `/admin(.*)` должен включать `frame-ancestors 'self'` вместо глобального `frame-ancestors 'none'` — Payload admin использует iframes для live preview

**Implementation hints:**

- CSP для `/admin(.*)` — пример конфигурации в `headers()`:

  ```js
  {
    source: '/admin(.*)',
    headers: [
      {
        key: 'Content-Security-Policy',
        value: [
          "default-src 'self'",
          "script-src 'self' 'unsafe-inline' 'unsafe-eval'",
          "style-src 'self' 'unsafe-inline' fonts.googleapis.com",
          "font-src 'self' fonts.gstatic.com data:",
          "img-src 'self' data: blob:",
          "connect-src 'self'",
          "frame-src 'self'",
          "frame-ancestors 'self'",
          "worker-src blob: 'self'",
        ].join('; '),
      },
    ],
  }
  ```

- Стабы коллекций — минимальный валидный экспорт: `import type { CollectionConfig } from 'payload'; export const Tags: CollectionConfig = { slug: 'tags', fields: [] }` — достаточно для компиляции и импорта в payload.config.ts
- Стабы глобалов: `import type { GlobalConfig } from 'payload'; export const Navbar: GlobalConfig = { slug: 'navbar', fields: [] }`
- Для db-postgres адаптера — проверить актуальный API по документации: в v3 может быть `pool: { connectionString }` или `url:` — использовать то, что соответствует текущей версии пакета
- onInit получает `payload` как аргумент: `onInit: async (payload) => { const count = await payload.count({ collection: 'users' }); if (count.totalDocs === 0) { ... } }`
- generateAPIKey не является встроенной функцией Payload — использовать `crypto.randomBytes(32).toString('hex')` из Node.js `crypto` модуля
