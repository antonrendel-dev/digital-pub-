# Task 10: Agent API Route (/api/articles)

Перед выполнением задачи загрузи:

## Description

Создаём `app/api/articles/route.ts` — кастомный Next.js route handler, который затеняет авто-генерируемый Payload REST эндпоинт `/api/articles`.

Задача двойная:

- `GET` — публичный фид статей (делегирует в `payload.find()` для опубликованных статей, чтобы избежать 405 от Next.js)
- `POST` — защищённый эндпоинт для агентов: извлекает `Authorization: users API-Key <token>`, валидирует через `payload.auth({ headers })`, проверяет `role === 'agent'`, затем создаёт статью через `payload.create()`

Ключевое ограничение: **НЕЛЬЗЯ использовать `overrideAccess: true`** — enforcement должен идти через нативный access control Payload, а не обходить его.

## What to do

1. Прочитать `payload.config.ts` (создан в Task 3) — понять структуру коллекции `articles` и пользователей

2. Создать `app/api/articles/route.ts` с двумя обработчиками:

   **GET — публичный фид:**

   ```ts
   export async function GET(req: Request) {
     const payload = await getPayload({ config })
     const result = await payload.find({
       collection: 'articles',
       where: { status: { equals: 'published' } },
     })
     return Response.json(result)
   }
   ```

   **POST — создание статьи агентом:**

   ```ts
   export async function POST(req: Request) {
     const payload = await getPayload({ config })

     // Аутентификация через Payload native auth
     const { user } = await payload.auth({ headers: req.headers })

     if (!user) {
       return Response.json({ error: 'Unauthorized' }, { status: 401 })
     }

     if (user.role !== 'agent') {
       return Response.json({ error: 'Forbidden' }, { status: 403 })
     }

     let body: unknown
     try {
       body = await req.json()
     } catch {
       return Response.json({ error: 'Invalid JSON' }, { status: 400 })
     }

     try {
       const article = await payload.create({
         collection: 'articles',
         data: body as Record<string, unknown>,
         req, // передаём req — Payload использует его для access control
       })
       return Response.json(article, { status: 201 })
     } catch (err: unknown) {
       const message = err instanceof Error ? err.message : 'Validation error'
       return Response.json({ error: message }, { status: 400 })
     }
   }
   ```

3. Убедиться что `getPayload` и `config` импортированы корректно:

   ```ts
   import { getPayload } from 'payload'
   import config from '@payload-config'
   ```

4. Проверить типизацию — `payload.auth({ headers })` возвращает `{ user: User | null }` в Payload v3

5. Запустить `npx tsc --noEmit` — TypeScript без ошибок

## Details

**Files to create:**

**Files to read first:**

- `payload.config.ts` — структура коллекций и конфигурация
- `payload/collections/articles.ts` — схема articles
- `payload/collections/users.ts` — схема users с полем role

**Dependencies:**

- Task 8 (Articles Collection) — коллекция articles должна существовать
- Task 9 (Users Collection & Roles) — коллекция users с полем role (agent/admin/sync) должна существовать

**Важные нюансы:**

- `app/api/articles/route.ts` затеняет Payload auto-generated `/api/articles`. Оба метода (GET и POST) должны быть реализованы явно — иначе Next.js вернёт 405
- `payload.auth({ headers })` в Payload v3 принимает `Headers` объект и возвращает `{ user: User | null }` — не использовать deprecated `payload.authenticate()`
- НЕ использовать `overrideAccess: true` — это обходит access control. Вместо этого передавать `req` в `payload.create()`, и Payload применит access rules из коллекции
- Формат Authorization header: `users API-Key <token>` — именно `users` (название коллекции) + `API-Key` + токен. Payload парсит его автоматически через `payload.auth()`
- При ошибке валидации Payload бросает исключение — ловить в try/catch и возвращать 400
- GET должен фильтровать только `status: published` — не отдавать draft/archived публично

**Edge cases:**

- Битый JSON в теле POST → `req.json()` бросает SyntaxError → вернуть 400
- Пустой body → Payload вернёт ValidationError (если title required) → 400
- Пользователь с role=sync пытается создать статью → 403 (только agent и admin могут)
- Payload может вернуть разные типы ошибок (ValidationError, Forbidden) — ловить общим catch и возвращать 400 для validation, но access-control ошибки Payload должен кидать сам через переданный `req`
