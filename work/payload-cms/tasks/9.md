# Task 9: App Routes + Globals Integration

## Required Skills

Перед выполнением задачи загрузи:

- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Интегрируем Payload Globals и объединяем источники статей. После завершения Tasks 4–7 (Collections + Globals полностью определены) Task 9 подключает компоненты и страницы к Payload Local API.

**Что меняется:**

1. `app/articles/page.tsx` — сейчас показывает только MDX-статьи из `lib/articles.ts`. После правки: мёрджит MDX-статьи + Payload-статьи (коллекция `articles`, status = 'published'), сортирует по `publishedAt` descending. MDX-статьи используют поле `date` из frontmatter как `publishedAt` (в `lib/articles.ts` схема имеет поле `publishedAt`, а фактические MDX-файлы могут называть его `date` — нужно отобразить).

2. `app/articles/[slug]/page.tsx` — сейчас берёт статью только из MDX. После правки: сначала ищет в Payload (`payload.find({ collection: 'articles', where: { slug: { equals: slug }, status: { equals: 'published' } } })`), если не найдена — fallback на MDX. Соответственно, `generateMetadata` тоже должна применять этот двойной поиск.

3. `components/Navbar.tsx` — сейчас `'use client'` компонент с хардкодом слогана `"Место, где встречаются хорошие люди"`. Navbar получает слоган через prop от серверного обёртки — нужно создать серверный wrapper, который вычитывает Navbar Global и пробрасывает данные в клиентский компонент. Либо разделить на ServerNavbar (SC) + ClientNavbar (CC).

4. `components/Footer.tsx` — сейчас чистый серверный компонент с хардкодом socialLinks и copyrightText. После правки: читает Footer Global из Payload, рендерит `socialLinks` (массив `{ platform, url }`) и `copyrightText`. Фалбэк на хардкодные значения если Global не настроен.

5. `components/LeftSidebar.tsx` — сейчас `'use client'` компонент, импортирует `SOCIAL_CHANNELS` из `lib/config.ts` для количества подписчиков. После правки: подписчики берутся из SocialChannels Global (`slug: 'social-channels'`, поле `channels[]` с `name`, `url`, `subscribers`). Аналогично Navbar — нужен серверный wrapper или серверный родитель, передающий данные через props.

Цель: любое изменение слогана, social links или copyright-текста в Payload Admin отражается на сайте в течение ≤1 секунды (через `revalidatePath('/', 'layout')` в afterChange хуках, которые определены в Task 7).

- Найти где используется `<Navbar>` (вероятно `app/layout.tsx` или `components/PageShell.tsx`) — заменить на `<NavbarServer>`.
- Найти где используется `<LeftSidebar>` — заменить на `<LeftSidebarServer>` (или передать `channels` prop сверху).

**Dependencies:**

- Task 4 (Tags Collection) — должна быть завершена (getPayload работает)
- Task 5 (Posts Collection) — должна быть завершена
- Task 6 (Articles Collection) — **критично**: поля `slug`, `status`, `publishedAt`, `content` коллекции `articles` должны существовать
- Task 7 (Globals) — **критично**: глобалы `navbar`, `footer`, `social-channels` должны быть определены
- `getPayload` из `payload` пакета (доступен после Task 3)

**Edge cases:**

- Payload Global может быть пустым при первом запуске (до заполнения через Admin UI) — всегда иметь хардкодный фалбэк для `slogan`, `copyrightText`, `socialLinks`, `channels`
- MDX-статьи в `lib/articles.ts`: фронтматтер-схема содержит поле `publishedAt`, но фактические `.mdx`-файлы могут использовать поле `date`. Проверить реальные файлы в `content/articles/` и при необходимости обработать оба варианта поля
- `LeftSidebar` — клиентский компонент, не может вызывать `getPayload()` напрямую. Данные из Global передаются через props из серверного wrapper'а
- `Navbar` — клиентский компонент (useState для search, menu), не может вызывать `getPayload()` напрямую. Только серверный wrapper передаёт данные через props
- Если Payload Local API недоступен (БД не поднята) — `try/catch` с фалбэком на хардкодные данные, `console.warn` в лог, страница не падает
- Payload-статьи с `status !== 'published'` не показываются в listing (публичный read-фильтр из access control в Task 6)

**Implementation hints:**

- `getPayload({ config })` импортируется из `payload`; `config` — дефолтный экспорт из `payload.config.ts`
- `payload.findGlobal({ slug: 'navbar' })` кэшируется Next.js автоматически (server component context = fetch dedup / React cache)
- Для мёрджа статей — выделить helper-функцию `mergeAndSortArticles(mdx, payload)` в отдельный файл или в `lib/articles.ts` для тестируемости
- Если Payload-статья содержит Lexical JSON в поле `content` — использовать `@payloadcms/richtext-lexical` для конвертации в HTML, или использовать `lexicalToHTML()` из пакета
