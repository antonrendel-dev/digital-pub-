# Task 1: Next.js 15 + React 19 Upgrade

## Required Skills

Перед выполнением задачи загрузи:

- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Апгрейд Next.js 14.2.35 → 15.x и React 18 → 19.x как изолированный, независимо деплоящийся шаг — до любой Payload-интеграции.

**Почему это первая задача:** Payload CMS v3 требует Next.js 15. Апгрейд делается отдельно, без Payload, чтобы изолировать breaking changes и проверить их на staging до того, как добавится сложность CMS.

**Главное breaking change Next.js 15:** `params` и `searchParams` в динамических роутах стали `Promise`-типизированными. В Next.js 14 можно было писать `const { category } = params` — в 15 это даст TypeError в runtime. Теперь нужно `const { category } = await params`. То же самое касается `generateMetadata({ params })` в каждом файле.

Затронуты **5 файлов** с динамическими роутами:

1. `app/vacancies/[category]/page.tsx` — params: `{ category: string }`
2. `app/vacancies/[category]/[slug]/page.tsx` — params: `{ category: string; slug: string }`
3. `app/resumes/tag/[tagSlug]/page.tsx` — params: `{ tagSlug: string }`
4. `app/articles/[slug]/page.tsx` — params: `{ slug: string }`
5. `app/post/[id]/page.tsx` — params: `{ id: string }`

В каждом файле нужно исправить и **page-компонент**, и **`generateMetadata`**.

После задачи: `npm run build` выходит 0, все существующие unit-тесты проходят, staging отдаёт 200 по всем динамическим роутам.

## What to do

1. **Обновить `package.json`:**
   - `next`: `"14.2.35"` → `"^15.0.0"` (или последняя стабильная 15.x)
   - `react`: `"^18"` → `"^19.0.0"`
   - `react-dom`: `"^18"` → `"^19.0.0"`
   - `@types/react`: `"^18"` → `"^19.0.0"`
   - `@types/react-dom`: `"^18"` → `"^19.0.0"`
   - `eslint-config-next`: `"14.2.35"` → совпадает с версией `next` (например `"^15.0.0"`)
   - Запустить `npm install` для обновления `package-lock.json`

2. **Исправить `app/vacancies/[category]/page.tsx`:**
   - Тип `Props`: изменить `params: { category: string }` → `params: Promise<{ category: string }>`
   - В `generateMetadata`: добавить `await` перед `params`, т.е. `const { category } = await params`
   - В `CategoryPage`: добавить `await` перед `params`, т.е. `const { category } = await params`

3. **Исправить `app/vacancies/[category]/[slug]/page.tsx`:**
   - Тип `Props`: изменить `params: { category: string; slug: string }` → `params: Promise<{ category: string; slug: string }>`
   - В `generateMetadata`: `const { slug, category } = await params` (сейчас `params.slug` и `params.category` без деструктуризации — привести к единому стилю)
   - В `VacancyPage`: `const { category, slug } = await params`

4. **Исправить `app/resumes/tag/[tagSlug]/page.tsx`:**
   - Тип `Props`: изменить `params: { tagSlug: string }` → `params: Promise<{ tagSlug: string }>`
   - В `generateMetadata`: `const { tagSlug } = await params`
   - В `TagPage`: `const { tagSlug } = await params`

5. **Исправить `app/articles/[slug]/page.tsx`:**
   - Тип `Props`: изменить `params: { slug: string }` → `params: Promise<{ slug: string }>`
   - В `generateMetadata`: `const { slug } = await params`
   - В `ArticlePage`: `const { slug } = await params`

6. **Исправить `app/post/[id]/page.tsx`:**
   - Тип `Props`: изменить `params: { id: string }` → `params: Promise<{ id: string }>`
   - В `generateMetadata`: добавить `const { id } = await params`, затем использовать `id` вместо `params.id` (сейчас `params.id` напрямую в двух местах — заменить оба)
   - В `PostPage`: `const { id: rawId } = await params`

7. **Запустить `npm run build`** — убедиться что выход 0 (нет ошибок TypeScript и ошибок сборки)

8. **Запустить `npm test`** — убедиться что все существующие unit-тесты проходят

## TDD Anchor

Тесты пишем ДО внесения правок в файлы роутов. Сначала тесты падают (потому что файлы ещё не исправлены), потом — после правок — проходят.

Файл: `tests/routes/async-params.test.ts`

```
tests/routes/async-params.test.ts::category_page_awaits_params
tests/routes/async-params.test.ts::category_slug_page_awaits_params
tests/routes/async-params.test.ts::resumes_tag_page_awaits_params
tests/routes/async-params.test.ts::articles_slug_page_awaits_params
tests/routes/async-params.test.ts::post_id_page_awaits_params
tests/routes/async-params.test.ts::category_generate_metadata_awaits_params
tests/routes/async-params.test.ts::resumes_tag_generate_metadata_awaits_params
tests/routes/async-params.test.ts::post_id_generate_metadata_awaits_params
```

**Что проверять в тестах:**

Каждый тест использует статический анализ (читает исходник файла как строку и проверяет паттерн) или мокирует Next.js окружение и вызывает page-функцию с `Promise`-params:

- Передать `params` как `Promise.resolve({ category: 'smm' })` в `CategoryPage` → убедиться что функция не бросает TypeError и вызывает `getTagBySlug` с правильным аргументом
- Аналогично для каждого из 5 файлов и каждой `generateMetadata`
- Или: читать исходник файла, проверять что `params: Promise<` присутствует в типе и `await params` присутствует в теле функции — простая строковая проверка как TDD-якорь

Тест падает до правок (файлы ещё с `params: { ... }` без Promise), проходит после.

**Файлы для создания:**

**Текущий код — конкретные места для правки:**

В `app/post/[id]/page.tsx` особый случай: в `generateMetadata` `params.id` используется **без деструктуризации** на строке 29 (`const url = \`https://d-pub.ru/post/${params.id}\``). После правки — destructure вверху функции через `await`, затем использовать переменную `id`.

В `app/vacancies/[category]/[slug]/page.tsx` в `generateMetadata` поля `params.slug` (строка 16) и `params.category` + `params.slug` (строки 25-26) используются без деструктуризации. После правки — `const { category, slug } = await params` в начале функции.

**Edge cases:**

- `app/articles/[slug]/page.tsx` использует `getArticleBySlug` (MDX, синхронный вызов из файловой системы), а не async DB-запрос — но params всё равно нужно await-ить, это требование Next.js 15 независимо от того, async ли остальной код страницы
- `app/post/[id]/page.tsx` валидирует id через zod (`idSchema.safeParse`) — эта логика не меняется, просто `rawId` теперь берётся из `await params`
- Next.js 15 также сделал async `cookies()` и `headers()` — в данных файлах они не используются, поэтому дополнительных правок не требуется

**Implementation hints:**

- Если `npm install` выдаёт peer dependency конфликты между React 19 и другими пакетами — проверить `next-mdx-remote@5.0.0` (может требовать React 18). В этом случае обновить `next-mdx-remote` до последней версии совместимой с React 19 или добавить `--legacy-peer-deps` как временное решение
