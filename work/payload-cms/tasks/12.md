## Focus Areas (чеклист аудита)

### 1. getPayload() только в server context

- `getPayload()` вызывается только в server компонентах, API route handlers, afterChange hooks, server actions — НЕ в файлах с `'use client'`
- Каждый вызов `getPayload()` делается внутри async функции (не на module level)
- Import `{ getPayload } from 'payload'` присутствует только в серверных файлах
- `payload.config.ts` импортируется через `import config from '@payload-config'` (алиас), не по прямому пути

### 2. Consistent error handling в lib/\*

- `lib/tags.ts`: все функции имеют try/catch; в catch — `console.warn('[tags] ...')` и возвращают null/[] (не бросают)
- `lib/posts.ts`: аналогично — try/catch, `console.warn('[posts] DB unavailable ...')`, возвращают `{ posts: [], total: 0, totalPages: 0 }` или null
- Нет голых `throw` без обработки в публичных функциях lib/\*
- `app/api/articles/route.ts`: все error paths возвращают Response с правильным status (401, 400, 500) — нет unhandled promise rejections

### 3. Нет Prisma imports

- Нет `import { PrismaClient }` или `from '@prisma/client'` в `app/`, `lib/`, `components/`
- `lib/prisma.ts` удалён (или его импорты нигде не используются)
- `prisma/seed.ts` удалён (не используется)
- `scripts/migrate-tags.ts` и `scripts/migrate-posts.ts` могут импортировать Prisma (они одноразовые миграционные скрипты — это нормально)
- `scripts/sync-telegram.ts` не использует Prisma (заменён на Payload REST API)

### 4. afterChange hook correctness

- Каждый hook в `payload/collections/*.ts` импортирует `revalidatePath` из `'next/cache'`
- Tags collection: hook вызывает ``revalidatePath(`/vacancies/${doc.slug}`, 'page')`` (с реальным slug документа, НЕ с шаблоном Next.js роута) + `revalidatePath('/', 'layout')`
- Posts collection: hook вызывает `revalidatePath('/vacancies/[category]', 'page')` для основной категории поста + `revalidatePath('/vacancies', 'layout')`
- Articles collection: hook вызывает `revalidatePath('/articles', 'layout')` + ``revalidatePath(`/articles/${doc.slug}`, 'page')``
- Globals (Navbar, Footer, SocialChannels, SiteSettings): каждый Global вызывает `revalidatePath('/', 'layout')`
- Hook функции объявлены как `async` (или корректно возвращают Promise)
- Hook не вызывает `revalidatePath` с undefined/null значениями — slug guard проверен

### 5. FeedPost type alignment

- `lib/postUtils.ts` — тип `FeedPost` отражает поля Payload (не Prisma): `tags` — массив объектов `{ id, slug }` (не строк), `channelUsername` и `telegramMessageId` присутствуют, поля в camelCase соответствуют Payload field names
- `lib/posts.ts` маппинг: Payload возвращает `relationship` поля как объекты — убедиться что `FeedPost.tags` собирается корректно (не `undefined`)
- `getPrimaryCategorySlug` в `postUtils.ts` работает с новой структурой tags (объекты `{ slug }`, не строки)

### 6. Access control correctness

- `payload/collections/users.ts`: `create: () => false` (нет self-registration)
- `payload/collections/articles.ts`: `create` разрешён для `agent` и `admin`; `update`/`delete` — только `admin`; публичный `read` возвращает только `{ status: { equals: 'published' } }` для неавторизованных
- `payload/collections/posts.ts`: `create` разрешён для `sync` и `admin`; `update`/`delete` — только `admin`
- `payload/collections/tags.ts`: только `admin` может создавать/редактировать/удалять
- `app/api/articles/route.ts`: auth проверяется через `payload.auth({ headers })` без `overrideAccess: true`; токен из заголовка `Authorization: users API-Key <token>` корректно передаётся в Payload

### 7. sync-telegram.ts безопасность

- API Key (`PAYLOAD_API_KEY`) не логируется в error messages (только status code и URL)
- `Authorization` header не появляется в console.error/console.warn
- Деградация на 409: пост пропускается без инкремента счётчика ошибок
